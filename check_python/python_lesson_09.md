# 第 9 讲：模块、包与代码拆分

## 这一讲的目标

学完这一讲，你应该能做到：

- 理解什么是模块
- 会使用 `import`
- 知道标准库和第三方库的区别
- 理解为什么项目要拆文件
- 能为一个小项目设计简单目录结构

这一讲非常关键，因为后面你做 Agent 项目时，不可能把所有代码都写进一个文件。

---

## 1. 什么是模块？

在 Python 里，一个 `.py` 文件通常就是一个模块。

例如：

```text
math_tools.py
main.py
config.py
```

这些都可以被别的 Python 文件导入使用。

---

## 2. 最简单的导入

假设有文件 `math_tools.py`：

```python
def add(a, b):
    return a + b
```

另一个文件 `main.py`：

```python
from math_tools import add

print(add(2, 3))
```

这就是最基础的模块复用。

---

## 3. `import` 的常见写法

### 写法 1

```python
import math
print(math.sqrt(9))
```

### 写法 2

```python
from math import sqrt
print(sqrt(9))
```

你现在只要能看懂这两种写法就够了。

---

## 4. 标准库和第三方库

### 标准库

Python 自带的库，比如：

- `math`
- `json`
- `os`

### 第三方库

需要额外安装的库，比如：

- `requests`
- `pandas`
- `openai`

后面学 API 调用时，你会接触第三方库。

---

## 5. 为什么要拆文件？

如果所有代码都堆在一个文件里，常见问题有：

- 很难读
- 很难维护
- 很难复用

例如你做一个简单 Agent，小项目也可能包含：

- 主程序入口
- 配置
- API 调用
- 消息构造
- 工具函数

这些放在不同文件里会更清晰。

---

## 6. 一个简单项目结构示例

```text
agent_project/
  main.py
  config.py
  agent.py
  api_client.py
  utils.py
```

你可以这样理解：

- `main.py`：程序入口
- `config.py`：配置项
- `agent.py`：核心逻辑
- `api_client.py`：接口调用
- `utils.py`：通用小工具

---

## 7. 什么时候该拆，什么时候不该过度拆？

如果只是一个很小的练习文件，没必要一开始拆很多层。

但当你开始出现这些情况时，就应该考虑拆分：

- 一个文件超过很多逻辑块
- 同一段代码被多个地方复用
- 配置、主流程、工具函数混在一起

当前阶段的原则可以简单一点：

- 先能跑
- 再清晰
- 再逐步拆分

---

## 8. 为什么模块化对 Agent 开发重要？

因为后面你会经常想把这些逻辑拆开：

- 消息拼装
- 模型调用
- 结果解析
- 工具执行
- 配置读取

如果不拆，你会越来越难调试。

---

## 9. 常见错误

### 错误 1：文件名和标准库重名

例如你自己创建 `json.py`，可能会影响导入真正的 `json` 库。

### 错误 2：一开始过度设计

小练习还没跑起来，就拆成很多层，反而让自己更乱。

### 错误 3：不知道主入口在哪里

一个项目至少要有一个明显的入口文件。

## 11. 本讲小结

这一讲最重要的是：

- 一个 `.py` 文件通常就是一个模块
- `import` 让代码可以复用
- 项目拆分是为了清晰和可维护
- 小项目不必过度拆，但真实项目一定要逐步模块化
